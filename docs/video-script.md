### 第 4 章：任务调度系统（5:00 - 7:00）

#### 4.1 长任务问题（约 30 秒）

- **内容要点**
  - 问题：FFmpeg 导出 30 分钟视频可能需要 5-10 分钟
  - 如果用同步 API 调用，前端会阻塞
  - 如果直接开线程，进度怎么推？取消怎么做？崩溃怎么处理？

#### 4.2 TaskManager 设计（约 50 秒）

- **内容要点**
  - 展示任务生命周期：

  ```
  create_task(type, payload)  ->  start_task(id)
                                     |
                                     v
                              [Worker Thread]
                                     |
                              progress_cb(percent)
                                     |
                                     v
                    ┌──────────────────────────────────┐
                    |         task:progress event      |
                    |  task:completed / task:failed    |
                    └──────────────────────────────────┘
  ```

  - 任务类型枚举（展示代码片段）：
    - EXPORT_VIDEO, EXPORT_AUDIO, EXPORT_SUBTITLE, EXPORT_VTT
    - SILENCE_DETECTION, FILLER_DETECTION, ERROR_DETECTION
    - FULL_ANALYSIS, TRANSCRIPTION, WAVEFORM_GENERATION
    - PLUGIN_INSTALL, MODEL_DOWNLOAD, PROXY_GENERATION

  - `main.py` 中注册 handler 的方式：

  ```python
  self._task_manager.register_handler(
      TaskType.EXPORT_VIDEO, self._handle_export_video
  )
  ```

- **旁白参考**
  - "所有耗时操作都通过 TaskManager 管理。前端创建任务、启动任务，后端在独立线程中执行，通过事件推送进度。"
  - "目前注册了 14 种任务类型，每种任务有对应的 handler 函数。"

#### 4.3 优先队列与并发控制（约 50 秒）-- 技术亮点

- **内容要点**
  - 任务分两类：

  ```python
  HEAVY_TASKS = {EXPORT_VIDEO, EXPORT_AUDIO, TRANSCRIPTION, SILENCE_DETECTION}
  LIGHT_TASKS = {WAVEFORM_GENERATION, PROXY_GENERATION}
  ```

  - 并发控制策略：
    - Heavy 任务：信号量 = 1（串行执行，避免 GPU/CPU 资源竞争）
    - Light 任务：信号量 = 3（最多 3 个并发，主要是 I/O 操作）
  - 使用 `PriorityQueue` + 原子计数器保证同优先级 FIFO

  - 架构图：

  ```
  [Task Queue: PriorityQueue]
         |
    [Dispatcher Thread]
       /          \
  Heavy(sem=1)   Light(sem=3)
      |              |
  [Thread Pool]  [Thread Pool]
  ```

- **旁白参考**
  - "任务分为重任务和轻任务。重任务比如视频导出，需要独占 GPU 或 CPU，所以用信号量限制同时只能跑一个。轻任务比如波形生成，主要是磁盘 I/O，可以同时跑三个。"
  - "所有任务通过优先队列调度，用原子计数器保证同优先级先进先出。"

#### 4.4 取消与竞态处理（约 20 秒）

- **内容要点**
  - 每个任务关联一个 `threading.Event` 作为取消信号
  - 任务 handler 在关键节点检查 `cancel_event.is_set()`
  - "Double-check pattern"：获取信号量之后再次检查任务是否已取消，防止执行已取消的任务

### 第 5 章：数据模型与持久化（7:00 - 8:30）

#### 5.1 Pydantic v2 frozen 模型（约 50 秒）

- **内容要点**
  - 展示核心模型定义：

  ```python
  class Segment(BaseModel, frozen=True):
      id: str
      version: int = 1
      type: SegmentType  # subtitle | silence
      start: float
      end: float
      text: str = ""
      words: list[Word] = Field(default_factory=list)
  ```

  - 解释 `frozen=True` 的意义：
    - 所有模型实例不可变（immutable）
    - 修改操作返回新实例：`segment.model_copy(update={...})`
    - 避免多线程状态竞争
    - 数据变更可追踪（version 字段）

  - 展示模型层级：
    ```
    Project
    ├── ProjectMeta (name, created, modified)
    ├── MediaInfo (path, duration, resolution, ...)
    ├── TranscriptData
    │   └── Segment[] (subtitle / silence)
    │       └── Word[] (word, start, end, confidence)
    ├── AnalysisData
    │   └── AnalysisResult[]
    └── EditDecision[] (action, status, priority)
    ```

- **旁白参考**
  - "所有数据模型都使用 Pydantic v2 的 frozen 模式，意味着创建之后就不能修改。"
  - "听起来不方便，但它解决了一个大问题：多线程安全。后台线程正在读一个 Segment 的时候，主线程不可能把它改掉。"
  - "如果需要修改，就创建一个新实例。Pydantic 的 model_copy 方法让这件事很简单。"

#### 5.2 项目文件结构（约 25 秒）

- **内容要点**
  - 每个项目保存为独立目录：
    ```
    data/projects/<name>/
    ├── project.json      # 所有业务数据
    └── waveform.json     # 波形峰值数据
    ```
  - `project.json` 就是 `Project` 模型的 JSON 序列化
  - 一个 JSON 文件包含全部状态，便于调试和迁移

#### 5.3 不可变模型 vs 可变操作（约 30 秒）

- **内容要点**
  - ProjectService 负责所有"修改"操作：
    - `add_silence_results()` -> 返回新的 `project` 字典
    - `update_edit_decision()` -> 返回新的 `project` 字典
    - `confirm_all_suggestions()` -> 返回新的 `project` 字典
  - 每次 API 调用都返回最新的完整 Project 状态给前端
  - 前端拿到新状态后替换本地响应式数据
  - "单向数据流，不可变数据，非常 Vue"

#### 6.1 "文本即剪辑"的交互隐喻（约 1 分钟）

- **内容要点**
  - 核心设计哲学："像编辑文档一样剪辑视频"
  - 交互映射：
    - 文档编辑 -> 视频剪辑
    - 选中文本按 Delete -> 标记删除该片段
    - Ctrl+Z -> 撤销
    - Ctrl+F -> 搜索替换
    - 行高亮 -> 选中对应视频片段
  - 为什么这样设计：
    - "每个人都有用文本编辑器的经验"
    - "时间轴编辑器的学习成本远高于文本编辑器"
    - "字幕本身就是视频内容的一维投影"
  - 演示：在编辑器中选中一行按 Delete，同时视频播放器跳转到对应位置

- **旁白参考**
  - "我认为 Milo-Cut 最好的设计决策不是某个算法，而是交互隐喻的选择：文本即剪辑。"
  - "时间轴编辑器功能强大，但学习成本高。而每个人都会用文本编辑器 -- 删一行字，按一下 Delete。"
  - "字幕本质上就是视频内容的一维投影。你不需要理解时间轴，只需要像编辑文档一样编辑字幕，视频自然会跟着变。"

#### 6.2 macOS .app Bundle 路径逃逸修复（约 1 分钟）

- **内容要点**
  - 问题背景：
    - macOS .app 实际是一个目录（`Milo-Cut.app/Contents/MacOS/...`）
    - PyInstaller 打包后，`sys.executable` 指向 `.app` 内部
    - 如果用相对路径（`../data/`），会"逃逸"出 `.app` 到用户桌面
  - 修复方案（展示代码）：

  ```python
  def _fix_macos_path():
      if sys.platform != "darwin" or not getattr(sys, "frozen", False):
          return
      shell = os.environ.get("SHELL", "/bin/zsh")
      result = subprocess.run(
          [shell, "-l", "-c", "echo $PATH"],
          capture_output=True, text=True, timeout=5,
      )
      if result.returncode == 0:
          os.environ["PATH"] = result.stdout.strip()
  ```

  - 额外处理：.app 启动时不加载 shell profile，导致 ffmpeg/uv 不在 PATH 中
  - 解决：启动时读取用户的 shell profile 注入 PATH

- **旁白参考**
  - "这是一个很典型的跨平台踩坑。macOS 的 .app 本质是个目录，打包后用相对路径读写文件会逃逸到外面去。"
  - "而且 .app 启动时不加载你的 shell profile，所以 ffmpeg 和 uv 都不在 PATH 里。"
  - "解决方案是在启动时手动读取用户的 zshrc，把 PATH 注入进去。简单粗暴，但有效。"

#### 6.3 便携模式：一个标记文件改变一切（约 45 秒）

- **内容要点**
  - 问题：同学之间想共享工具，但数据存在用户目录下（`~/.local/share/milo-cut/`）
  - 解决：在应用目录下放一个 `.portable` 空文件
  - 检测逻辑：

  ```python
  # paths.py (simplified)
  if Path(app_dir / ".portable").exists():
      data_dir = Path(app_dir / "data")
  else:
      data_dir = Path.home() / ".local" / "share" / "milo-cut"
  ```

  - 效果：整个应用 + 数据可以放在 U 盘里携带

- **旁白参考**
  - "便携模式可能是最简单的功能，但用户反馈最好。只需要在应用目录下创建一个叫 `.portable` 的空文件，所有数据就会存在应用目录下，而不是系统目录。"
  - "这样你把整个文件夹拷到 U 盘里，插到另一台电脑上就能用，零配置。"

#### 6.4 编码器注册表与自动降级（约 45 秒）

- **内容要点**
  - 问题：用户选了 NVENC（NVIDIA 硬件编码），但机器上没有 NVIDIA 显卡
  - 解决：编码器注册表 + 运行时检测 + 自动降级

  ```python
  # 启动时检测可用编码器
  video_codec, fallback_msg = get_fallback_codec(ffmpeg, video_codec)
  if fallback_msg:
      self._emit("encoder:fallback", {
          "requested": original_codec,
          "fallback": video_codec,
      })
  ```

  - 前端收到 `encoder:fallback` 事件后弹出提示："NVENC 不可用，已切换为 libx264"

- **旁白参考**
  - "视频导出支持多种编码器，包括硬件编码。但问题是，你不能假设用户的机器上一定有 NVIDIA 显卡。"
  - "所以我在导出前会检测编码器是否可用，不可用就自动降级到软件编码，并通过事件通知前端。用户不会看到一个报错，只会看到一个温和的提示。"

#### 6.5 规则引擎 vs 深度学习：为什么选规则（约 45 秒）

- **内容要点**
  - 填充词检测和口误检测使用规则引擎，不是深度学习
  - 规则引擎实现：
    - 填充词：正则匹配 + 自定义词表
    - 口误：触发词列表（"等一下"、"我重新说"、"不对"）
    - 重复检测：n-gram 余弦相似度 > 0.85
  - 为什么选规则：
    - "填充词检测的规则是明确的，不需要训练"
    - "规则引擎更可控、更可调试、更易定制"
    - "用户可以自己改词表，不需要重新训练模型"
    - "深度学习留给 ASR 转录，那是它擅长的事"

- **旁白参考**
  - "填充词检测，很多人第一反应是用 NLP 模型。但实际上，'嗯'、'那个'、'就是说'这些词的规则非常明确。"
  - "规则引擎的好处是：可控、可调试、可定制。用户在设置里加一个词，立刻就能生效，不需要下载模型或重新训练。"
  - "我的原则是：简单的问题用简单的方案。深度学习留给语音转录，那是它真正不可替代的地方。"
