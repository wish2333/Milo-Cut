## 5. 技术债务与架构风险

### 5.1 已知问题

| 问题 | 影响 | 优先级 |
|------|------|--------|
| `main.py` 版本号硬编码 "0.1.0" | 版本显示不一致 | 高 |
| `media_hash` 字段始终为空 | 无法检测源文件是否被移动/覆盖/篡改 | 高（v1.1.0 修复） |
| `proxy_path` 字段未使用 | 大文件（4K/数 GB）直接加载可能导致前端 OOM | 中 |
| `speaker` 字段未使用 | 多人对话场景无法区分说话人 | 低 |
| `Word` 模型未在 UI 中使用 | 词级编辑能力未释放 | 低 |

### 5.2 架构风险审查

#### 快速导出（`-c copy`）的技术陷阱

- **问题**：H.264/H.265 等高压缩比编码中，`-c copy` 要求切割点必须在 I 帧（关键帧）上。若切割点落在 P/B 帧，会导致绿屏、马赛克、花屏、音画不同步。语音静音段的边界几乎不可能刚好对齐 I 帧。
- **方案 A（推荐）**：Smart Rendering -- 仅在切割点所在的 GOP（图像组）重新编码，其余部分 copy。对 FFmpeg 命令行拼接要求极高，需要精确计算 GOP 边界。
- **方案 B（降级）**：仅提供纯音频的快速导出，或在 UI 上明确警告"快速导出可能存在画面卡顿，仅供内部预览"。
- **结论**：移至 v1.3.0，预估 1-2 周实现 Smart Rendering 方案。

#### Undo/Redo 与状态管理的前置依赖

- **问题**：当前前端状态分散在 13 个 composables 中，无统一状态管理。在复杂时间线编辑器（波形 + 片段合并/拆分 + 分析结果 + 编辑决策）中，没有不可变数据流和 Command 模式，强行实现 Undo/Redo 将导致灾难性的状态不一致。
- **结论**：Pinia 重构必须作为 Undo/Redo 的前置任务，两者合并为 v1.1.0 P0，总预估 2-3 周。

#### ASR/VAD 的依赖膨胀与模型管理

- **问题**：FunASR / faster-whisper 依赖 PyTorch（数 GB），模型文件数百 MB 至数 GB。若作为默认依赖打包，应用体积臃肿，无 GPU 设备可能启动失败。
- **结论**：AI 模型必须设计为**按需下载加载（Lazy Load / Plugin）**模式。"本地模型管理 UI"从 P2 提升为 v1.2.0 的 P0 前置任务，用户首次使用转写功能时才触发模型和依赖环境的下载配置。

#### 大文件内存/性能风险

- **问题**：`proxy_path` 未使用，前端直接加载 4K 原片绘制波形和预览。数 GB 素材可导致浏览器内存溢出（OOM）页面崩溃。
- **结论**：波形数据已预渲染为 JSON（`generate_waveform()`），但视频预览仍需代理文件机制。列为 v1.3.0 优化项。
