| 功能 | 影响 | v1.1.0 优先级 |

| 文稿对齐 | 无法自动匹配讲稿与实际录制 | 低 |

### 3.4 步骤控制器

PRD 设计了 5 步流程控制器（导入 -> 转写 -> 分析 -> 编辑 -> 导出），v1.0.0 采用简洁的三页导航（欢迎 -> 工作区 -> 导出），**有意简化**，非遗漏。

**决策理由**：当前三页流程已覆盖完整用户路径，步骤控制器增加 UI 复杂度但不增加核心价值。v1.1.0 不恢复此设计。

## 4. v1.1.0 迭代规划

### 4.1 优先级定义

- **P0（必须）**：核心交互补齐 + 崩溃防护 + 技术债务清理
- **P1（重要）**：状态管理优化 + Undo/Redo + 体验增强
- **P2（增强）**：交互细节打磨

### 4.2 v1.1.0 目标功能

#### P0 -- 核心交互与安全防护

> **v1.1.0 核心主题：先补安全，再加交互，最后固架构。**

| 功能 | 说明 | 预估工作量 | 依赖 |
|------|------|-----------|------|
| EditSummaryModal 接入导出流程 | 组件已开发完成（含测试），仅需挂载到 ExportPage，导出前弹出删除摘要确认对话框 | 0.5-1 天 | 无 |
| 项目文件自动保存 | 编辑操作后自动保存到 `project.json`，防崩溃/刷新丢数据。需防抖（debounce 300-500ms）避免频繁写入 | 2-3 天 | 无 |
| 媒体丢失重链接（Relink Media） | 打开项目时检测 `media.path` 是否存在，不存在则弹出重定位对话框；同时计算 `media_hash`（SHA-256）校验源文件是否被覆盖 | 2-3 天 | 无 |
| 版本号统一 | 修复 `main.py` 硬编码 "0.1.0"，统一从 `pyproject.toml` 读取或三处同步更新 | 0.5 天 | 无 |
| 字幕叠加预览 | 播放器底部实时字幕叠加显示，跟随当前播放时间。纯前端实现（HTML/CSS 同步 `currentTime`），无需代理文件或转码 | 2-3 天 | 无 |
| FFmpeg 管理与设置页 | 新增设置页面，集成 FFmpeg 检测/下载/版本切换功能。参考 ff-intelligent-neo 项目的 `FFmpegSetup` 组件：6 级优先链检测（用户指定 > 打包 > 本地 > 平台路径 > PATH > static_ffmpeg）、版本发现与切换、一键下载。Milo-Cut 已依赖 `static_ffmpeg` 包，可直接复用 | 1 周 | 无 |

#### P1 -- 状态管理与 Undo/Redo

| 功能 | 说明 | 预估工作量 | 依赖 |
|------|------|-----------|------|

| 前端状态管理评估（Pinia） | 评估是否需要将 11 个 composables 迁移到 Pinia store。当前 composable 架构已合理（模块级单例实现全局状态），Pinia 主要价值为 DevTools 调试支持。建议 Undo/Redo 实现后再评估是否需要 Pinia | 2-3 天评估 + 实施 | Undo/Redo 完成后 |

#### 从 v1.1.0 移出的功能

| 功能 | 原计划 | 移出原因 | 移至版本 |
|------|--------|---------|---------|

| VAD 增强静音检测 | v1.1.0 P1 | 同 ASR，依赖模型管理基础设施 | v1.2.0 |
| 重复句检测 | v1.1.0 P1 | 依赖 sentence-transformers 模型，需模型管理基础设施 | v1.2.0 |

### 5.2 架构风险审查

#### 快速导出（`-c copy`）的技术陷阱

- **问题**：H.264/H.265 等高压缩比编码中，`-c copy` 要求切割点必须在 I 帧（关键帧）上。若切割点落在 P/B 帧，会导致绿屏、马赛克、花屏、音画不同步。语音静音段的边界几乎不可能刚好对齐 I 帧。
- **方案 A（推荐）**：Smart Rendering -- 仅在切割点所在的 GOP（图像组）重新编码，其余部分 copy。对 FFmpeg 命令行拼接要求极高，需要精确计算 GOP 边界。
- **方案 B（降级）**：仅提供纯音频的快速导出，或在 UI 上明确警告"快速导出可能存在画面卡顿，仅供内部预览"。
- **结论**：移至 v1.3.0，预估 1-2 周实现 Smart Rendering 方案。

#### Undo/Redo 的实现路径

- **方案 A（推荐）**：Command 模式 + 快照栈。利用 Pydantic 不可变模型（`frozen=True`），每次编辑操作前保存 `Project` 对象的深拷贝到历史栈。实现简单，与现有 composable 架构兼容。
- **方案 B**：Pinia + Command 模式。迁移状态到 Pinia store 后实现 Undo/Redo。增加 DevTools 支持，但迁移成本高。
- **结论**：采用方案 A，先在 composables 上实现 Undo/Redo。如后续评估 Pinia 价值足够，可作为独立优化项。

#### ASR/VAD 的依赖膨胀与模型管理

- **问题**：FunASR / faster-whisper 依赖 PyTorch（数 GB），模型文件数百 MB 至数 GB。若作为默认依赖打包，应用体积臃肿，无 GPU 设备可能启动失败。
- **结论**：AI 模型必须设计为**按需下载加载（Lazy Load / Plugin）**模式。"本地模型管理 UI"从 P2 提升为 v1.2.0 的 P0 前置任务，用户首次使用转写功能时才触发模型和依赖环境的下载配置。

#### FFmpeg 管理风险

- **问题**：Milo-Cut 的所有核心功能（静音检测、视频导出、音频导出、波形生成）都依赖 FFmpeg/FFprobe。当前仅通过 `settings.json` 手动配置路径，无检测、无下载、无版本管理。用户首次安装时极可能遇到"找不到 FFmpeg"的报错。
- **方案**：参考 ff-intelligent-neo 项目实现 6 级优先链检测（用户指定 > 打包 > 本地 > 平台路径 > PATH > static_ffmpeg）。Milo-Cut 已依赖 `static_ffmpeg` 包，可直接复用其下载能力。新增设置页面承载 FFmpegSetup 组件。
- **结论**：纳入 v1.1.0 P0，预估 1 周。

#### 大文件内存/性能风险

- **问题**：前端直接加载 4K 原片播放视频。数 GB 素材在浏览器中可能导致解码卡顿。
- **现状评估**：当前工作区播放器仅占 40% 宽度（`w-2/5 min-w-[400px]`），浏览器内置缩放处理高分辨率源文件。波形已预渲染为 JSON，完全独立于视频。字幕叠加是纯前端 HTML/CSS。因此**当前架构下代理文件非必需**。
- **未来需求**：`proxy_path` 字段已预留，`PreviewPlayer.vue` 已在消费。仅在以下场景需要代理文件：浏览器无法原生解码的格式（ProRes、DNxHD）、需要全分辨率流畅拖拽的场景。
- **结论**：代理文件生成列为 v1.3.0 优化项，当前架构已足够。
