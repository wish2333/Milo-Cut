### 1.1 项目背景

在短视频和在线课程爆发式增长的背景下，口播类视频的后期制作中存在大量重复性体力劳动。录制一小时的口播素材，剪辑师往往需要花费 2-3 小时手动聆听、标记停顿、删除口误和口头禅。这一过程不仅耗时，而且极易因疲劳导致遗漏或误判。

现有的专业剪辑工具（如 Premiere Pro、达芬奇）功能过于重量级，侧重于精剪和特效包装，缺乏针对"素材清洗"这一特定环节的自动化支持。而一些在线工具虽然提供了静音检测功能，但依赖云端处理，存在隐私泄露风险，且对中文口播场景（口头禅识别、口误触发词检测）支持不足。

Milo-Cut 的定位是"录完第一步打开的 AI 清洗工具"——不是完整剪辑软件，而是专注于将粗录素材快速清洗为可剪辑的干净素材。核心理念是"像改文档一样剪视频"：用户在字幕文本编辑器中删除一段文字，就等于删除了对应的视频片段。

### 1.2 用户画像

- **口播博主 / 知识博主**：录制 30-90 分钟口播视频，口误多、停顿多、重复表达频繁
- **课程录制者 / 教培团队**：45-120 分钟录屏加讲解，经常讲错重讲、存在长时间停顿
- **播客 / 访谈剪辑师**：60-180 分钟多人对话，需要快速定位重点片段、去除废话
- **实习同事 / 同学**：在教育实习场景中需要快速处理录课视频，但不具备专业剪辑技能

## 2. 痛点与动机

### 使用前（痛点）

- **素材清洗耗时占总创作时间 40% 以上**：一小时的粗录视频，手动标记停顿、口误、口头禅需要 2-3 小时
- **工具碎片化**：静音检测用一个工具，字幕编辑用另一个，导出又需要第三个工具，流程割裂
- **中文口播场景缺乏优化**：现有工具大多基于英文设计，无法识别"嗯""啊""然后就是"等中文口头禅，也无法检测"不对重来""说错了"等口误触发词
- **隐私顾虑**：在线工具需要上传视频到云端，对于企业培训、内部课程等敏感内容存在合规风险

### 使用后（预期/实际改善）

- **粗剪时间预计减少 50%+**：自动检测静音、口头禅、口误，用户只需确认而非手动标记
- **一站式工作流**：导入视频 -> 自动分析 -> 文本式编辑 -> 多格式导出，全流程在单一工具内完成
- **中文口播深度优化**：内置可配置的口头禅词表和口误触发词规则，覆盖中文口播常见场景
- **本地优先处理**：核心功能全部在本地完成，音视频数据不离开用户电脑

## 4. 产品方案

### 4.1 方案概述

Milo-Cut 采用"字幕即剪辑"的核心交互范式，将视频编辑转化为文本编辑体验。用户导入粗录视频和对应的 SRT 字幕后，系统通过三层分析引擎（规则层、相似度层、LLM 层）自动标记问题片段，用户在字幕编辑器中确认或拒绝建议，最终导出干净的视频和字幕文件。

### 4.2 核心功能设计

#### 功能一：三层智能分析引擎

**做什么**：自动检测视频中的静音段、口头禅、口误触发词和重复表达，为用户提供删除建议。

**设计原因**：纯静音检测只能找到"安静的地方"，但无法识别"有声废话"（如口头禅、口误后的废段）。三层架构（规则 -> 相似度 -> LLM）让用户可以根据需求选择分析深度，同时保证规则层的高置信度（0.85-0.95）。

**关键设计决策**：
- 规则层使用正则匹配和 FFmpeg silencedetect，确定性高、速度快
- 分析结果默认标记为 pending（黄色），用户确认后才变为 confirmed（红色），避免误删
- 导出前强制弹出摘要弹窗，显示"将删除 X 段，共 Y 分钟（占总量 Z%）"，三道保险防止误操作

#### 功能二：字幕编辑器与波形视图

**做什么**：提供类似文档编辑器的字幕编辑体验，配合波形可视化和视频预览，实现"所见即所得"的剪辑操作。

**设计原因**：传统时间线编辑器学习成本高，对于只做素材清洗的用户来说过于复杂。字幕编辑器将"删除视频片段"简化为"删除一行文字"，大幅降低使用门槛。

**关键设计决策**：
- 静音段以独立行穿插在字幕段之间，灰色背景标注"静音 N.Ns"，用户可拖拽调整边界
- 波形视图使用 Canvas 渲染，支持点击跳转和鼠标滚轮缩放时间范围
- 字幕行和波形视图双向同步：点击字幕行跳转视频，播放时自动高亮当前字幕行
- 采用 Vue 3 依赖注入（provide/inject）机制管理时间轴指标，实现波形组件间的解耦

#### 功能三：静音-字幕统一时间轴与联动编辑

**做什么**：将机器检测出的静音段和用户导入的字幕段融合在同一条时间轴上，让用户在一个界面里同时看到"哪里有话、哪里没话、哪里是废话"，并能对两种片段进行统一的编辑操作。

**设计原因**：这是 Milo-Cut 最核心的产品设计难点。市面上的工具通常把静音检测和字幕编辑做成两个独立模块——用户先在一个界面看静音检测结果，再切到另一个界面看字幕，靠脑子记住"这段静音前后说了什么"来判断该不该删。这种割裂的体验在素材较长时尤其痛苦：一小时的视频可能有上百个静音段，用户根本记不住每个停顿的上下文。

**核心设计思路——"像改文档一样剪视频"**：

统一时间轴的设计目标是：让用户看到的不是两条分离的数据流，而是一份连续的"视频内容文档"。在这份文档里，字幕段就是正文，静音段就是段落间的空白。删除一个静音段就像删除文档中的多余空行一样自然。

具体来说，设计上做了三个关键决策：

- **同一条时间轴，同一套操作**：静音段和字幕段在界面上按时间顺序交替排列，静音段以灰色背景行穿插在字幕行之间，标注"静音 2.3 秒"。用户不需要切换视图，一眼就能看到完整的视频节奏——这段话之后停了多久、下一段话从哪里开始。删除操作也完全统一：不管是删一行字幕还是删一段静音，都是"选中 -> 确认删除"，用的是同一套操作逻辑和同一套状态管理（待确认 -> 已确认 / 已拒绝）。

- **智能检测而非机械切分**：静音检测不是简单地把所有安静的地方都标记出来。设计了一条三阶段的检测流水线来提高准确性：首先，把检测到的静音区间两端各收缩一小段边距，避免把语音的起振和衰减部分误切；然后，用字幕的时间范围做保护——如果一个静音段紧贴着字幕的边界，就把静音段往里收，确保不会侵入字幕的有效语音范围；最后，为每个静音段自动生成一条"建议删除"的编辑决策，但如果同一位置已经有用户之前的手动标记，就跳过不重复建议。这三步保证了检测结果"开箱即用"，用户大部分情况下直接确认就行，不需要手动调整边界。

- **独立但可联动的边界调整**：静音段和字幕段是各自独立的数据实体——拖动一个静音段的边界不会自动推移相邻字幕的时间戳。这个设计是有意为之的：自动检测只是建议，最终的剪辑精度由用户掌控。用户可以微调任何一个静音段的起止时间来精确控制删除范围，调整后系统会自动同步更新对应的编辑建议。同时，批量操作也做了来源隔离——"清除所有静音建议"只影响机器检测的结果，不会动用户手动标记的编辑，两类操作互不干扰。

#### 功能四：多格式导出系统

**做什么**：支持导出 MP4（精确/快速双模式）、SRT 字幕、EDL、FCPXML、Premiere XML、OTIO 等多种格式，兼容主流剪辑软件。

**设计原因**：用户完成素材清洗后，通常需要在专业剪辑软件（达芬奇、Premiere Pro、FCP）中继续精剪。多格式导出确保 Milo-Cut 能无缝融入现有工作流。

**关键设计决策**：
- 精确导出使用 FFmpeg 重新编码，帧级精度；快速导出使用 -c copy，关键帧级但速度更快
- OTIO（OpenTimelineIO）作为通用中间格式，支持音频过渡时长独立控制
- 导出前自动检测异常情况（删除段超过 40%、单段超过 60 秒、连续删除超过 3 段），弹窗确认

### 4.3 系统架构

```
Milo-Cut Desktop App
|
|-- Python Backend (PyWebVue Bridge)
|   |-- main.py (MiloCutApi)     -- @expose API 入口
|   |-- TaskManager              -- 统一异步任务管理
|   |-- Core Services
|   |   |-- ffmpeg_service       -- FFmpeg/ffprobe 封装
|   |   |-- analysis_service     -- 三层分析引擎
|   |   |-- export_service       -- 视频导出（精确/快速）
|   |   |-- export_timeline      -- 时间线导出（EDL/XML/OTIO）
|   |   |-- subtitle_service     -- SRT 解析与生成
|   |   |-- media_server         -- 本地 HTTP 媒体流服务
|   |   |-- project_service      -- 项目文件管理
|   |-- Models (Pydantic v2)     -- 类型安全的数据模型
|
|-- Vue 3 Frontend
    |-- pages/                   -- WelcomePage / WorkspacePage / ExportPage
    |-- components/
    |   |-- waveform/            -- WaveformCanvas / WaveformEditor / SegmentBlocksLayer
    |   |-- workspace/           -- TranscriptRow / SilenceRow / SuggestionPanel / Timeline
    |   |-- export/              -- EncodingSettings / PreviewPlayer
    |-- composables/             -- useProject / useEdit / useExport / useSegmentEdit 等
```

**前后端通信**：PyWebView 提供桌面窗口壳，通过自定义 Bridge 层（@expose 装饰器 + 事件系统）实现 Python 后端与 Vue 前端的双向通信。后端方法通过 `window.pywebview.api` 暴露给前端，前端通过 `call("method_name", ...args)` 调用，返回统一的 `{success, data, error}` 响应信封。

**异步任务系统**：所有耗时操作（静音检测、分析、波形生成、导出）通过 TaskManager 统一管理。前端创建任务后获得 task_id，通过事件系统（task:progress / task:completed / task:failed）接收进度更新。事件流转依赖 PyWebView 的 tick 机制——JS 端每 50ms 轮询一次事件队列，实现跨线程通信。

## 5. 开发过程

### 5.1 规划与调研阶段

项目启动前，我系统调研了现有的视频粗剪工具生态：

- **竞品分析**：研究了 Descript（云端依赖、中文支持弱）、auto-editor（纯静音检测、无文本编辑）、Premiere Pro（功能过重、学习成本高）等工具的优劣
- **需求验证**：与同学和实习同事交流视频剪辑痛点，确认"素材清洗耗时"是最普遍的抱怨
- **技术选型**：评估了 PyWebView（轻量、原生感强、Python 生态丰富）vs Electron（体积大、内存占用高）vs Tauri（Rust 学习曲线），最终选择 PyWebView + Vue 3 的组合
- **PRD 编写**：产出完整的 PRD 文档，定义 P0/P1/P2 功能优先级和验收标准

### 5.2 MVP 实现（Phase 0 - Phase 1）

MVP 阶段聚焦于最小可用闭环：

1. **项目骨架搭建**（5月14日）：基于 PyWebVue 框架初始化项目，建立 Python 后端 + Vue 3 前端的通信桥接层
2. **核心工作流实现**（5月14-15日）：实现静音检测、SRT 字幕导入、字幕编辑器、视频/SRT 导出的完整链路
3. **分析 UI 与编辑交互**（5月15日）：实现规则引擎分析、字幕编辑操作（确认/拒绝/批量操作）、编辑决策管理
4. **波形编辑器**（5月15日）：新增 WaveformCanvas、TimeMarksLayer、WaveformEditor 组件，实现波形可视化和交互式编辑

MVP 在两天内完成了从零到可用的跨越，验证了技术方案的可行性。

### 5.3 迭代与增强

#### 迭代一：稳定性修复（5月15-16日）

MVP 暴露了大量交互层面的 Bug：时间戳编辑异常、文本重叠、按钮冻结、状态切换逻辑不一致等。通过多轮代码审计（audit-report 系列），系统性地修复了 7 个 HIGH 级别 Bug 并实现了 4 个缺失功能。

**触发原因**：快速原型阶段的代码质量债务
**解决方案**：引入结构化审计流程——先产出 audit-report 发现问题，再制定 audit-plan 逐一修复
**成效**：编辑器交互稳定性显著提升，核心工作流（导入 -> 分析 -> 编辑 -> 导出）可完整走通

#### 迭代二：导出系统完善（5月16日）

导出系统经历了从"能用"到"好用"的关键迭代：

- 实现 ExportPage 完整页面，包含编码设置、预览播放器和时间线导出
- 重写 EDL/XML 导出以兼容达芬奇和 Premiere Pro
- 修复导出编码设置不生效、路径解析错误、预览无音量控制等多个 Bug
- 改进 GPU 编码器自动检测

**触发原因**：实际导出测试中发现与专业剪辑软件的兼容性问题
**解决方案**：深入研究各剪辑软件的工程文件格式规范，针对性调整导出逻辑
**成效**：导出的 EDL/XML 文件可在达芬奇和 Premiere Pro 中正确导入

#### 迭代三：音频过渡与波形系统（5月16-17日）

这一轮迭代解决了音频处理和波形显示的深层问题：

- 实现 FFmpeg xfade 视频过渡和音频交叉淡入淡出系统
- 添加 OTIO（OpenTimelineIO）导出支持，音频过渡时长可独立控制
- 修复波形数据的项目隔离问题（不同项目共享同一波形文件的 Bug）
- 实现波形持久化、事件驱动加载和缓存失效机制
- 添加 Toast 通知系统，改善操作反馈体验

**触发原因**：多项目场景下波形数据串扰、音频过渡时长不灵活
**解决方案**：采用 per-project 波形文件存储架构，前端使用 cache-busting 时间戳强制刷新波形 URL
**成效**：波形显示可靠、音频过渡可控、用户体验连贯

## 7. 亮点与挑战

### 7.1 技术挑战

- **挑战一：PyWebView 跨线程通信**
  - **问题**：PyWebView 的 `evaluate_js` 只能在主线程调用，但耗时操作（FFmpeg、分析）必须在后台线程执行，如何实现双向通信？
  - **解决方案**：设计了 tick-based 事件循环——JS 端每 50ms 调用一次 `tick()` 方法，该方法从 Python 端的事件队列中取出待发送事件并执行，同时处理一个排队中的任务。后台线程通过 `run_on_bridge()` 将工作调度到主线程执行。
  - **收获**：深入理解了桌面应用中 UI 线程与工作线程的协作模式，以及事件驱动架构的设计原则

- **挑战二：波形数据的项目隔离与缓存一致性**
  - **问题**：多个项目可能引用同一媒体文件，波形数据存储路径设计不当导致项目间数据串扰；波形重新生成后前端 URL 未更新，显示旧数据。
  - **解决方案**：将波形文件存储路径从基于 media_hash 改为基于 project 目录，确保物理隔离；前端在波形重新生成后附加 cache-busting 时间戳参数，强制浏览器重新请求。
  - **收获**：缓存一致性是桌面应用中容易被忽视的问题，需要在架构层面提前规划

- **挑战三：两种不同来源的数据如何在同一时间轴上和谐共存**
  - **问题**：静音段来自机器检测（FFmpeg），字幕段来自用户导入（SRT），它们的生命周期完全独立。核心矛盾在于：检测阶段可以用字幕范围做保护、避免静音段侵入语音区域，但用户后续手动调整边界时，这种保护不会自动重新执行——用户可能把一个静音段的边界拖进了字幕范围，造成逻辑冲突。更棘手的是，删除操作的来源多样（机器建议、用户手动、字幕裁剪），需要在统一的状态管理中区分"谁提的建议"，才能支持"只撤回机器建议、保留手动标记"这类精细操作。
  - **解决方案**：在架构层面做了两个关键决策——第一，**检测时充分防御，编辑时不过度约束**：检测流水线通过边距收缩和字幕保护裁剪，确保初始生成的静音段就已经是合理的边界；后续用户手动调整时，系统不做强制约束（比如禁止拖进字幕区域），而是信任用户的判断。这比全局实时校验更实用，因为用户的微调意图系统无法完全预判。第二，**统一的编辑决策模型，按来源隔离操作**：所有删除建议（不管来自机器还是用户）走同一个状态机管理，但每条建议都标记了来源。这样"清除所有机器建议"只删除特定来源的记录，不影响用户手动做的标记，两类操作互不干扰。
  - **收获**：在多数据源融合的场景中，"创建时充分防御 + 编辑时信任用户 + 操作时按来源隔离"的策略比全局约束更务实。过度约束会让系统变得僵硬（用户无法灵活调整），而完全不约束又会导致数据混乱——关键是在哪个环节施加多大的约束力

- **挑战四：多格式导出的兼容性**
  - **问题**：不同剪辑软件（达芬奇、Premiere Pro、FCP）对 EDL/XML 工程文件的格式要求存在细微差异，初始实现无法被部分软件正确导入。
  - **解决方案**：深入研究各软件的格式规范，针对性调整导出逻辑；引入 OTIO 作为通用中间格式，利用 OpenTimelineIO 库的标准化能力降低适配成本。
  - **收获**：文件格式兼容性需要实际测试验证，不能仅依赖文档描述

### 7.2 设计亮点

- **"字幕即剪辑"交互范式**：将复杂的视频时间线编辑简化为类似文档编辑的体验，用户删除一行文字就等于删除一段视频。这一设计大幅降低了非专业用户的使用门槛，是产品最核心的差异化价值。
- **三层分析引擎架构**：规则层（高置信度）-> 相似度层（需确认）-> LLM 层（可选增强），用户可以根据需求选择分析深度。这种分层设计既保证了核心功能的可靠性，又为后续扩展预留了空间。
- **三道误删防护机制**：导出前强制摘要弹窗 + 编辑状态分级（pending/confirmed/rejected）+ 异常检测警告（40%/60s/连续3段），从多个维度防止误操作导致的内容丢失。
- **静音-字幕统一时间轴架构**：这是 Milo-Cut 区别于其他视频粗剪工具的核心架构决策。传统做法是把静音检测和字幕编辑做成两个独立模块，用户在两个视图之间来回切换。Milo-Cut 选择把两种不同来源的数据（机器检测的静音段 + 用户导入的字幕段）融合到同一条时间轴上，用同一套操作逻辑管理。这个决策使得"删除一段静音"和"删除一行字幕"在用户侧完全等价——都是在同一条列表里选中一行、点确认删除。它也是"字幕即剪辑"范式能够成立的技术基础：如果不是所有类型的片段都在同一条轴上，用户就不可能获得"像改文档一样剪视频"的体验。

### 7.3 开发流程亮点

- **AI 辅助的 VibeCoding 开发模式**：充分利用 AI 工具进行代码生成、架构设计和问题诊断，将开发重心从"写代码"转移到"做决策"——明确需求、设计方案、把控质量、验证结果。这种模式在两天内实现了从零到 MVP 的跨越。
- **结构化审计驱动迭代**：每轮迭代都遵循"audit-report（发现问题）-> audit-plan（制定方案）-> 实施修复 -> 记录文档"的闭环流程，确保问题被系统性地发现和解决，而非随机修补。
- **文档先行的开发习惯**：在编码前完成 PRD、设计规范、组件规范等文档，为后续开发提供了清晰的参考系。开发过程中持续维护 record 文档，记录每次变更的原因和影响。

## 8. 项目复盘

### 做得好的地方

- **快速验证了技术方案的可行性**：PyWebView + Vue 3 + Python 的混合架构在桌面应用场景下表现良好，前后端通信桥接层设计合理，为后续迭代奠定了坚实基础
- **MVP 交付速度超出预期**：两天内完成从零到可用的跨越，证明了 AI 辅助开发模式在独立项目中的高效性
- **审计驱动的质量保障机制有效**：多轮审计发现了大量交互层面的问题，并通过结构化的修复流程逐一解决

### 可改进的地方

- **初期代码质量债务较多**：快速原型阶段为了赶进度，部分代码缺乏充分的错误处理和边界检查，导致后续需要大量修复工作。未来应在 MVP 阶段就建立基本的代码质量标准
- **测试覆盖不够充分**：虽然编写了单元测试，但集成测试和端到端测试覆盖不足。部分交互层面的 Bug（如波形缓存失效）需要实际运行才能发现
- **跨平台适配尚未验证**：当前仅在 Windows 11 上开发和测试，macOS 和 Linux 的兼容性尚未验证

### 关键收获

- **独立开发全栈桌面应用的完整经验**：从需求分析、产品设计、技术选型到编码实现、测试迭代、文档维护，完整走通了一个桌面应用的全生命周期
- **AI 辅助开发的方法论沉淀**：探索了 VibeCoding 模式下如何高效利用 AI 工具，同时保持对架构决策和代码质量的把控
- **"先让它能用，再让它好用"的迭代哲学**：不追求一步到位的完美设计，而是通过 MVP 快速验证核心假设，再通过多轮迭代持续优化用户体验
