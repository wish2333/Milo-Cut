# Milo-Cut 项目经历

## 1. 项目概述

| 维度 | 内容 |
|------|------|
| 项目名称 | Milo-Cut |
| 项目类型 | AI驱动的视频预处理工具 |
| 我的角色 | 产品经理（兼独立开发者） |
| 开发周期 | 2026年3月 - 至今（v0.1.0 - v1.3.0） |
| 技术栈 | Python + Vue 3 + PyWebView + FFmpeg |
| 目标用户 | 需要进行口述视频剪辑的同学、实习同事 |

Milo-Cut是一款面向学术场景的视频预处理工具，核心解决口述报告视频（课程汇报、论文答辩、学术会议发言）的粗剪效率问题。通过AI语音识别自动检测静音段、口误和填充词，将原本1-2小时的手动剪辑工作缩短至10-15分钟。

## 2. 痛点与动机

### 2.1 用户痛点发现

在研究生期间，我观察到身边同学在处理口述视频时普遍面临以下困扰：

- **录制冗长**：30分钟的汇报视频，有效内容可能只有20分钟，大量静音停顿和口头禅（"嗯"、"那个"、"就是说"）需要手动删除
- **工具门槛高**：专业剪辑软件（达芬奇、Premiere）学习成本高，同学大多只会用剪映等简单工具，但这些工具缺乏批量处理能力
- **重复劳动多**：每次录制都需要重新剪辑，缺乏可复用的工作流

### 2.2 优化前后对比

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| 粗剪耗时 | 1-2小时/30分钟视频 | 10-15分钟 |
| 工具要求 | 需要学习专业剪辑软件 | 开箱即用，无需学习 |
| 问题识别 | 逐帧手动检查 | AI自动标记问题片段 |
| 批量处理 | 不支持 | 支持多项目批量导出 |

## 3. 主要职责

作为产品经理，我负责：

- **需求调研**：访谈5-10位同学，了解口述视频剪辑的真实痛点
- **产品定义**：确定核心功能范围（语音转录、静音检测、口误识别、批量导出）
- **优先级排序**：按用户价值和技术可行性排列版本迭代计划
- **用户体验设计**：设计工作流（导入→转录→分析→确认→导出），确保操作简洁
- **版本管理**：规划4个主要版本的功能边界，控制范围蔓延
- **用户验证**：邀请同学试用，收集反馈并迭代优化

## 4. 产品方案

### 4.1 产品定位

**一句话定位**：口述视频的AI粗剪助手

**核心价值主张**：
- 不替代专业剪辑，只做"粗剪"——删除静音、口误、填充词
- 降低剪辑门槛——无需学习专业软件，开箱即用
- 提升效率——AI自动识别问题，用户只需确认或调整

### 4.2 核心功能设计

**功能1：AI语音转录（v1.2.0）**

用户痛点：手动听写30分钟视频的字幕，耗时且容易遗漏

产品方案：
- 集成多个ASR引擎（faster-whisper、Qwen-ASR、MLX-ASR）
- 支持CPU和GPU推理，适应不同硬件环境
- 转录结果可编辑，用户可修正识别错误

设计决策：
- 采用插件化架构，而非绑定单一引擎——因为不同引擎在不同场景下表现不同，用户应有选择权
- 支持自定义模型目录——部分同学网络环境不佳，需要离线使用预下载的模型

**功能2：智能问题检测（v1.1.0）**

用户痛点：即使有字幕，仍需逐句检查是否有口误、重复、填充词

产品方案：
- 规则引擎自动检测：静音段、填充词（"嗯"、"那个"、"就是说"）、重复语句
- 检测结果以时间轴形式展示，用户可快速定位问题片段
- 支持自定义填充词列表——不同方言区的口头禅不同

设计决策：
- 采用规则引擎而非深度学习——因为填充词检测的规则相对明确，规则引擎更可控、更易调试
- 提供"原片/剪后"切换预览——让用户直观看到剪辑效果

**功能3：批量导出（v1.3.0）**

用户痛点：每次录制都需要重新剪辑，缺乏批量处理能力

产品方案：
- 支持多项目批量导出，一次性处理多个视频
- 实时进度追踪，用户可随时取消
- 支持多种导出格式（视频、音频、SRT字幕、OTIO/EDL时间线）

设计决策：
- 采用优先队列任务管理——重任务（导出）串行执行，轻任务（波形生成）并发执行，避免资源竞争
- 懒加载代理生成——代理文件在首次预览时生成，而非导入时，提升首次打开速度

**功能4：便携模式与跨平台（v1.2.1-v1.2.2）**

用户痛点：同学之间需要共享工具，但每个人的电脑环境不同

产品方案：
- 便携模式：通过`.portable`标记文件激活，数据存储在应用目录下，可U盘携带
- 跨平台支持：Windows和macOS（含Apple Silicon）
- 数据目录可配置：模型文件可独立存储，支持跨实例复用

### 4.3 用户旅程

```
导入视频 → AI转录 → 智能检测 → 用户确认 → 批量导出
   ↓          ↓          ↓          ↓          ↓
  拖拽或    自动生成    自动标记    逐个确认    选择格式
  选择文件   字幕       问题片段    或批量确认   一键导出
```

## 5. 开发过程

### 5.1 版本规划策略

采用"小步快跑"的迭代策略，每个版本聚焦一个核心问题：

| 版本 | 主题 | 核心目标 | 用户价值 |
|------|------|----------|----------|
| v1.2.0 | AI转录 | 实现语音转文字 | 基础能力补齐 |
| v1.2.1 | 数据管理 | 便携模式+模型管理 | 便于分享和离线使用 |
| v1.2.2 | macOS支持 | Apple Silicon适配 | 覆盖更多用户设备 |
| v1.3.0 | 性能优化 | 批量导出+内存优化 | 提升处理效率 |

### 5.2 需求优先级决策

在v1.3.0版本中，面临多个需求的优先级选择：

| 需求 | 用户价值 | 技术复杂度 | 优先级 |
|------|----------|------------|--------|
| 批量导出 | 高（用户强烈反馈） | 中 | P0 |
| 内存优化 | 高（100+片段会崩溃） | 高 | P0 |
| 懒加载代理 | 中（提升首次打开速度） | 低 | P1 |
| 打包体积优化 | 低（不影响功能） | 低 | P2 |

决策依据：
- 批量导出是用户反馈最多的需求，优先实现
- 内存优化是稳定性问题，必须解决
- 懒加载代理是体验优化，可后续迭代
- 打包体积优化是技术债务，不影响用户体验

### 5.3 用户反馈收集

通过以下方式收集用户反馈：

- **同学试用**：邀请5位同学试用，收集操作流程和功能建议
- **问题记录**：建立问题跟踪表，记录用户遇到的bug和困惑
- **版本复盘**：每个版本发布后，回顾哪些功能用户用得多、哪些用得少

## 7. 亮点与挑战

### 7.1 产品亮点

**亮点1：精准的用户定位**

没有试图做成"万能剪辑软件"，而是聚焦"口述视频粗剪"这一细分场景。这使得产品功能简洁、学习成本低，同学之间容易传播。

**亮点2：渐进式功能设计**

从基础的语音转录开始，逐步添加问题检测、批量导出、跨平台支持。每个版本都有明确的用户价值，避免了"大而全"的功能堆砌。

**亮点3：用户参与式设计**

邀请同学参与试用和反馈，而非闭门造车。例如，"便携模式"功能就是基于同学"想在实验室和宿舍之间共享工具"的需求而设计的。

### 7.2 产品挑战

**挑战1：AI转录的准确性**

问题：ASR引擎在学术场景下的识别准确率不够高，特别是专业术语和口音较重的用户。

应对：
- 提供转录结果编辑功能，允许用户修正错误
- 支持多个ASR引擎，用户可选择最适合自己口音的引擎
- 设计"填充词自定义"功能，适应不同方言区的口头禅

**挑战2：性能与用户体验的平衡**

问题：100+片段的视频导出时内存爆炸，导致应用崩溃。

应对：
- 优先解决稳定性问题（O(1)内存优化）
- 采用懒加载策略，避免首次打开时的等待
- 提供实时进度追踪，让用户知道应用在工作

**挑战3：跨平台兼容性**

问题：Windows和macOS的路径规范、打包方式不同，功能表现不一致。

应对：
- 设计便携模式，减少对系统路径的依赖
- 针对macOS .app Bundle专门处理路径逃逸问题
- 在两个平台上分别进行功能验证

### 7.3 过程亮点

**亮点1：数据驱动的优先级决策**

通过用户反馈和使用数据，而非个人偏好来决定功能优先级。例如，批量导出功能的优先级提升，正是因为多位同学反馈"需要处理多个视频"。

**亮点2：小步快跑的迭代策略**

每个版本聚焦一个核心问题，避免了"大版本"的风险。例如，v1.2.0只做ASR转录，v1.2.1再做数据管理，v1.2.2做跨平台，v1.3.0做性能优化。

**亮点3：用户教育与预期管理**

明确告知用户这是"粗剪助手"而非"专业剪辑软件"，避免了用户对功能范围的误解。同时，通过"原片/剪后"切换预览，让用户直观看到AI剪辑的效果。

## 8. 项目复盘

### 8.1 做得好的地方

- **需求洞察准确**：从同学的真实痛点出发，产品定位清晰
- **功能设计克制**：聚焦"粗剪"场景，避免功能蔓延
- **迭代节奏稳定**：4个版本按时交付，每个版本都有明确价值
- **用户参与充分**：同学参与试用和反馈，产品更贴近真实需求

### 8.2 可改进的地方

- **用户规模有限**：目前主要在同学圈使用，缺乏更大范围的验证
- **数据收集不足**：缺乏系统化的使用数据收集，难以量化用户行为
- **竞品分析薄弱**：对同类工具的分析不够深入，可能存在功能重叠

### 8.3 经验收获

- **细分场景的价值**：聚焦细分场景比做大而全的产品更容易成功
- **用户反馈的重要性**：早期用户反馈能显著提升产品方向的准确性
- **技术可行性的考量**：产品经理需要理解技术约束，避免提出不可能实现的需求
- **迭代思维的价值**：小步快跑比大版本发布更可控、风险更低
