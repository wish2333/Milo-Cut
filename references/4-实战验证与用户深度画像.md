## 分析

在您已经完成了竞品分析、技术选型（Python + PyWebVue + FFmpeg）以及明确了“本地 AI 粗剪预处理器”这一核心定位 后，下一步的调查应从**理论设想**转向**实战验证**与**用户深度画像**。

以下是建议的四个关键调查方向：

------

### 1. 核心算法可行性专项调查（算法侧）

目前产品最核心的差异化在于从“静音剪辑”进化到“语义粗剪”。您需要验证 ASR 模型在真实口播环境下的表现。

- **口误重说（Re-read）模式识别**：针对“说错 → 停顿 → 重说”或“说错 → 纠正（不对、重来） → 重说”的典型口播场景，收集 50-100 个真实样本进行测试。
- **FunASR vs. Whisper 专项评测**：在嘈杂环境、地方口音、中英混读三种场景下，对比两个引擎的词级时间戳（Word-level timestamp）精度。精度直接决定了“剪辑点”是否准确。
- **语义相似度降噪**：调研 `sentence-transformers` 等模型在识别“重复表达”上的准确率，验证是否能自动折叠冗余信息。

### 2. 用户痛点与工作流深度访谈（需求侧）

竞品分析告诉了我们“别人做了什么”，但没有告诉我们“用户最痛苦的那个点”是什么。

- **素材清洗占比调查**：针对您的目标用户（知识博主、课程录制者）进行访谈，确认他们在“选材”和“去废话”上究竟花费了多少时间。
- **“误删”容忍度测试**：用户更倾向于“AI 自动全删，我找回误删”还是“AI 标记建议，我手动确认”？这决定了您产品的交互逻辑（是全自动还是辅助式）。
- **本地隐私敏感度**：确认目标用户对“全本地化处理”的支付意愿是否显著高于云端工具（如 Descript），这关系到未来的定价策略。

### 3. 专业剪辑软件集成协议调查（工程侧）

由于您的定位是“第一步打开的预处理器”，与下游专业软件（PR/FCPX/剪映）的兼容性至关重要。

- **FCPXML & Premiere XML 深度兼容性**：调研如何通过 XML 传递“标记（Markers）”或“静音段折叠”信息，而不仅仅是简单的片段拼接。
- **无损切割技术验证**：深入调研 `LosslessCut` 所采用的流拷贝（Stream Copy）技术，验证在不同编码格式下，无损切割后生成的素材在 PR/FCPX 中的兼容性。

### 4. 性能与资源消耗基准测试（性能侧）

由于采用本地化 ASR 和 FFmpeg 批处理，硬件门槛是影响用户体验的关键。

- **显存与内存占用**：在 8GB/16GB 内存的普通笔记本上，处理 1 小时 4K 素材时的 ASR 速度与系统流畅度。
- **Webview 长时间线渲染**：利用 `PyWebVue` 开发时，调研在 Web 端渲染 2 小时以上超长音轨视图的性能瓶颈及优化方案。

## XML 导出方案验证

从 **XML 协议调研**开始，我们现在的目标是打通本工具与专业剪辑软件（如 Premiere Pro）之间的“信息桥梁” 。

这一步的关键在于如何将我们在软件中生成的 `edit_decisions.json` 标记，精准地映射为 Premiere Pro 能够识别的 **Markers（标记）** 。

------

### 📂 XML 协议调研执行路线

针对你提出的“确认 Pr 能正常显示建议删除标记”的目标，建议按以下步骤进行调研：

#### 1. 确定目标格式：Premiere XML (FCP 7 XML)

虽然 Premiere 属于 Adobe 系列，但它主要通过 **Final Cut Pro 7 XML (.xml)** 格式进行序列交换 。这是目前兼容性最广的格式，能够承载时间轴、剪辑点和标记信息 。

#### 2. 标记（Marker）的 XML 结构映射

你需要在 XML 文件中定位并构造 `<marker>` 标签。一个典型的 Premiere 兼容标记通常包含以下关键属性：

- 

  **`<name>`**：标记的名称（例如：`建议删除`） 。

- 

  **`<comment>`**：详细说明（例如：`ASR 识别为口误`） 。

- 

  **`<in>` / `<out>`**：标记在时间轴上的起始点和结束点（以帧数为单位） 。

- **`<color>`**：标记的颜色。建议使用**红色**来醒目提示删除建议。

#### 3. 核心实验步骤

为了快速验证可行性，工程侧建议执行以下操作：

1. **手动导样**：在 Premiere Pro 中手动创建一个序列，并在时间轴上添加一个红色的、带有“建议删除”注释的标记。

2. **导出参考**：通过 `文件 -> 导出 -> Final Cut Pro XML` 导出该序列。

3. **逆向分析**：用文本编辑器打开导出的 `.xml`，搜索 `marker` 关键字，观察其嵌套层级和参数格式。

4. 

   **编写脚本验证**：尝试用 Python 编写一个极简脚本，生成一个包含同样结构的 XML 文件，并重新导入 Pr，观察标记是否原样出现 。

------

### 🎯 调研交付目标

完成此项调研后，工程侧应产出以下成果：

- **标记模板**：一个经过验证的、可被 Python 动态填充的 XML 字符串模板。
- **帧率转换逻辑**：确定如何将 ASR 输出的秒级时间戳准确转换为 XML 所需的帧数（需考虑 23.976, 25, 30, 60 等常见帧率）。
- **兼容性清单**：确认除 Premiere 外，该 XML 是否也能被达芬奇（DaVinci Resolve）正常读取（这符合你之前的用户调研需求）。

## 附录：用户痛点与工作流深度访谈结果

- 用户想要自由地一遍录完稿子再对录音进行清洗，删除录错的和重复的内容，保持稳定的句间距，能够导入到精剪软件（主要是Pr和达芬奇）中继续剪辑
- 功能需要能够模块化，不是一键全流程，需要可以一步步方便地传递执行，并且能够跳过不需要进行的步骤或者是在某些步骤允许用户手动介入来完成
- 用户在素材清洗上面花费的时间往往多于录音所消耗的时间（包括边录音边清洗工作流中的清洗时间）
- 用户往往希望能够一键识别，标记出删除建议，并通过取消选择的方式来保留一些错误识别的结果
- 用户不需要有额外的经济支出和云端依赖（除了大语言模型功能以外），本项目基本决定保守开源
- 用户希望在软件中能够预览播放并直观地看到字幕，并能够便利地对字幕进行修改
