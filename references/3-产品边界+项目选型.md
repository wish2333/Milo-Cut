下面是评估结论：**建议做“本地 AI 粗剪预处理器”，不要做完整剪辑软件**。你们现有技术积累非常匹配这个方向，尤其是 `ff-intelligent-neo` 已经具备 FFmpeg、auto-editor、桌面 GUI、任务队列、预设、打包等基础能力；`PyWebVue` 可作为新项目/重构项目的桌面开发底座。([GitHub](https://github.com/wish2333/ff-intelligent-neo)) ([GitHub](https://github.com/wish2333/PyWebVue))

------

## 1. 项目边界建议

### 核心定位

做一款面向**口播、访谈、课程、直播录像**的本地 AI 粗剪工具：

> 导入粗录视频 → 自动转写 → 标记停顿、口误、重复、废话 → 用户确认 → 导出干净素材 / 字幕 / 剪辑工程文件。

这与你们已有构想一致：当前市场空白在于把**字幕、静音检测、语义识别**融合起来，服务“录完第一步打开的 AI 预处理器”场景。

------

## 2. 明确做什么、不做什么

### MVP 必做

| 模块         | 目标                                         |
| ------------ | -------------------------------------------- |
| 视频导入     | 支持 mp4/mov/mkv，提取音频，生成基础媒体信息 |
| 本地 ASR     | 生成带时间戳字幕，优先支持中文口播           |
| 静音检测     | 标记长停顿、空白段                           |
| 口头禅检测   | 嗯、啊、然后、就是、那个等可配置词表         |
| 口误检测     | “不对重来”“刚才说错了”“重新说”等规则识别     |
| 文本驱动粗剪 | 删除文本片段，对应删除视频时间段             |
| 预览与撤销   | 原片 / 剪后对比，删除片段可恢复              |
| 导出         | MP4、SRT、剪辑决策 JSON，后续再做 XML/FCPXML |

### MVP 不做

| 暂不做                 | 原因                                 |
| ---------------------- | ------------------------------------ |
| 多轨复杂时间线         | 会把项目拖成 Premiere/剪映 替代品    |
| 特效、包装、滤镜、美颜 | 偏成片制作，不是粗剪核心             |
| 自动一键成片           | 误判风险高，早期应“AI 建议 + 人确认” |
| 云端协作               | 与本地隐私定位冲突                   |
| 素材库管理             | 第二阶段再做                         |

## 3. 技术选型主方案

### 推荐主方案：Python + PyWebVue + Vue 3 + FFmpeg + FunASR/Whisper

这是最适合你们当前团队的路线。

| 层级       | 推荐                                    |
| ---------- | --------------------------------------- |
| 桌面壳     | PyWebView / PyWebVue                    |
| 前端       | Vue 3 + TypeScript + TailwindCSS        |
| 后端       | Python 3.11+                            |
| 包管理     | uv                                      |
| 前端包管理 | bun 或 npm                              |
| 视频处理   | FFmpeg / ffprobe                        |
| 静音检测   | FFmpeg silencedetect + auto-editor 思路 |
| ASR        | FunASR 优先，Whisper 备选               |
| 语义分析   | 规则引擎优先，LLM 作为增强              |
| 数据存储   | SQLite + JSON 工程文件                  |
| 打包       | PyInstaller onedir 优先                 |

理由：`ff-intelligent-neo` 已经是 Python + Vue + PyWebView 架构，具备批处理、FFmpeg 命令预览、auto-editor 自动剪辑、跨平台打包等能力，可直接复用大量工程经验。([GitHub](https://github.com/wish2333/ff-intelligent-neo)) `PyWebVue` 本身也是为 Python 开发者准备的 Python + Vue 桌面快速开发框架，支持 Python bridge、Vue 调用、PyInstaller 打包。([GitHub](https://github.com/wish2333/PyWebVue))

## 4. ASR 选型

### 方案 A：FunASR，推荐默认

适合中文口播、本地化、速度和部署友好。
建议用于 MVP。

优点：中文效果好，部署成本较低，适合国内用户。
缺点：生态和字幕编辑案例不如 Whisper 普及。

### 方案 B：faster-whisper

适合多语言、中英混合、生态成熟。

优点：稳定、社区成熟、支持 GPU/CPU。
缺点：中文标点、口语断句可能需要后处理。

### 方案 C：双引擎策略

MVP 先内置 FunASR，设置里允许切换 Whisper。
这是更稳妥的产品方案。

## 5. “废话/口误识别”不要一开始就全靠 LLM

建议采用三层结构：

### 第一层：确定性规则

用于高置信标记：

| 类型       | 示例                           |
| ---------- | ------------------------------ |
| 静音       | 超过 800ms / 1200ms / 2000ms   |
| 口头禅     | 嗯、啊、就是、然后、那个       |
| 重说触发词 | 不对、重来、重新说、刚才说错了 |
| 废弃片段   | “这段不要”“剪掉这段”           |

### 第二层：文本相似度

用于识别重复句、重复解释：

| 技术                                 | 用途                   |
| ------------------------------------ | ---------------------- |
| sentence-transformers / bge-small-zh | 句向量相似度           |
| difflib / rapidfuzz                  | 近似重复文本           |
| 时间邻近窗口                         | 判断是否是“说错后重说” |

### 第三层：LLM 辅助

只做“建议”，不直接删除：

| 功能       | 方式                     |
| ---------- | ------------------------ |
| 判断跑题   | 给出建议删除标签         |
| 章节划分   | 生成结构                 |
| 短视频切片 | 找高价值片段             |
| 粗剪评分   | 高价值 / 可疑 / 建议删除 |

这样可控、可解释，也更适合本地产品。

## 8. 开发路线

### 第 0 阶段：技术验证，1 个闭环

目标：验证“字幕 + 静音 + 删除导出”。

做：

1. 导入视频；
2. 提取音频；
3. ASR 出字幕；
4. 检测静音；
5. 用户勾选删除；
6. FFmpeg 导出 MP4。

不做复杂 UI，不做 LLM。

### 第 1 阶段：MVP

重点是“可用”。

功能：

1. 项目管理；
2. 字幕文本编辑器；
3. 静音、口头禅、口误标记；
4. 删除建议列表；
5. 预览；
6. MP4/SRT/JSON 导出。

### 第 2 阶段：增强版

功能：

1. 文稿对齐；
2. 重复句识别；
3. 章节生成；
4. FCPXML / Premiere XML；
5. 短视频切片建议。

### 第 3 阶段：产品化

功能：

1. 多套粗剪方案；
2. 本地模型管理；
3. 插件化规则；
4. 批量处理；
5. 商业版授权。

## 9. 备选技术方案

### 方案 A：沿用 PyWebVue，最推荐

适合你们现在的团队。

优点：开发快、Python 友好、可复用现有项目经验。
缺点：超复杂时间线性能不如原生/Electron。

结论：**MVP 首选。**

### 方案 B：Electron + Python 后端

优点：前端生态强，复杂 UI 更稳。
缺点：体积大，Python 通信和打包复杂。

结论：适合后期重 UI 产品，不适合一开始。

### 方案 C：Tauri + Rust + Python sidecar

优点：体积小、性能好。
缺点：Rust 学习成本高，团队 Python 优势无法充分发挥。

结论：暂不推荐。

### 方案 D：纯 Web 本地服务

FastAPI + Vue，用户浏览器访问 localhost。

优点：开发简单，调试方便。
缺点：桌面体验、文件权限、打包分发弱。

结论：适合原型，不适合最终桌面产品。

------

## 10. 最终建议

**新项目不要从零开始，也不要直接把 `ff-intelligent-neo` 改成 AI 剪辑软件。**

推荐做法：

1. 以 `PyWebVue` 作为新项目骨架；
2. 从 `ff-intelligent-neo` 迁移 FFmpeg、任务队列、路径管理、打包、auto-editor 相关经验；
3. 产品上独立命名，避免被用户理解为“FFmpeg 批处理工具的一个功能”；
4. 第一版只打穿：**导入 → 转写 → 标记 → 确认删除 → 导出**；
5. AI 能力先规则化、可解释化，LLM 后置。

一句话：
**技术路线选 Python + PyWebVue + Vue + FFmpeg + FunASR，产品边界守住“粗录清洗”，不要扩张成完整剪辑软件。**
